Project Vision & Architectural Strategy
Bridging the gap between raw TTS capabilities and
classroom-ready hardware.
The Core Bottleneck: Speak-AI currently provides natural speech
exclusively in English. For every other language—Hindi, Arabic,
Swahili, Kinyarwanda, Quechua—the system defaults to eSpeak-ng.
This produces robotic, mispronounced audio that actively disrupts the
language learning process in global classrooms.
The Strategic Fix:
Instead of bringing in massive architectural changes, the
project leverages the existing infrastructure in the codebase
(alt_tts_backends.py and tts_cache.py).
By integrating these modules into the project, a robust routing
pipeline is achieved that is strictly designed for low-resource
environments.

The Execution Path:

1)​ Data-Driven Dispatch: Routes text dynamically to the best offline
engine based on language requirements (Kokoro for core languages,
Piper for Arabic/Swahili, MMS for indigenous languages).
2)​ Hardware-Aware Defenses: Enforces dynamic RAM detection
and a sub-millisecond caching layer to actively prevent
out-of-memory crashes on 1GB classroom laptops.
3)​ Zero Silence UX: Uses asynchronous background loading with
immediate eSpeak placeholder, completely eliminating severe UI
freezes associated with heavy neural models.
4)​ Objective Validation: Scores every language integration against a
reproducible Phoneme Error Rate (PER) target to ensure true native
speaker accuracy.

Project Overview
Speak-AI is an advanced version of the traditional 'speak' activity developed by
Sugar Labs.
This new version replaces the traditional 'espeak' engine with an
82-million-parameter neural TTS model named 'Kokoro,' which provides
natural-sounding and human-like speech, an upgrade that is greatly beneficial to
children learning languages and improving reading skills.
The current version of 'Kokoro' supports eight languages: English, Spanish,
French, Hindi, Italian, Brazilian Portuguese, Japanese, and Mandarin.
This means there are hundreds of millions of children whose native language is
not among them: Arabic, Swahili, Kinyarwanda, Quechua, Guarani, and Aymara,
among many others.
This project extends the multilingual support of Speak AI to include these
underserved languages.
As natural-sounding speech is a priority for auditory learning, the quality of
pronunciation is the first priority of each decision made by this proposal.

Project Goals
1.​ Multilingual TTS Expansion: Enhance existing support for Spanish,
Brazilian Portuguese, Hindi, French, and Mandarin. Add six underserved
languages: Arabic, Swahili, Kinyarwanda, Quechua, Guarani, and Aymara.
2.​ Pronunciation Quality: Extend G2P, explore TTS backends for
languages unsupported by Kokoro, and develop a scored evaluation system
with test data for each language.
3.​ Performance Optimization: Investigate ONNX Runtime for CPU
inference speedups and integrate existing but unused audio cache to avoid
redundant inference for repeated audio.
4.​ Offline-First Model Management: Create ModelManager: check local
cache, validate SHA-256 hashes for downloads, and use existing but
unused audio cache to fall back to espeak if a model is unavailable to
ensure activity always produces audio, even without internet.
5.​ Community Validation: Implement three native speaker review
checkpoints (Weeks 2, 6, 11), using a scored rubric, involving native
speakers from the Sugar Labs mailing list, Kinyarwanda Wikimedians, and
OpenStreetMap contributors for Arabic.

Current Architecture
How Speak-AI works today:
If a child types something and hits enter, the following happens:
The Gtk.Entry widget from activity.py calls face.View.say(text), which calls
Speech.speak() from speech.py.
The system first checks if Kokoro is present.If it is,
_ensure_kokoro_lang() from activity.py chooses the appropriate G2P model for
the language of the selected voice, and KPipeline.call() from kokoro/pipeline.py
converts the text into IPA phonemes.
The phonemes, along with a voice embedding, are then converted into raw
24kHz audio by KModel.forward() from kokoro/model.py.
If Kokoro is not present, it goes straight to GStreamer espeak.
In both cases, it goes into a GStreamer pipeline, where it splits into two sinks:
autoaudiosink for speaker, and fakesink (16kHz S16LE) for the mouth animation
of the face widget.

The pipeline explained:
1.​ activity.py — GTK UI. Child writes to Gtk.Entry, which triggers
face.View.say(text)
2.​ speech.py — Routing. Speech.speak() determines whether to use Kokoro
or espeak. If Kokoro is present, it will call _ensure_kokoro_lang() to
change the language of the G2P model according to the language of the
voice selected
3.​ kokoro/pipeline.py — G2P + synthesis. KPipeline does text →
phonemes using either misaki (for English, Japanese, Chinese) or
espeak-ng (for everything else), then feeds phonemes and voice
embedding to KModel
4.​ kokoro/model.py — Neural network. KModel does phonemes → 24kHz
F32 WAV using the phoneme string and a 256-dimensional voice
embedding tensor
5.​ GStreamer — Audio is pushed using appsrc, split using tee to
autoaudiosink (speakers) and fakesink (mouth animation, S16LE 16kHz)

Why this approach is the right one for 90 hours
One thing to consider with a multilingual TTS project is whether the time is best
spent writing new language support from scratch, or whether the time is best
spent making the existing code work. This proposal takes the obvious route: the
building blocks are in the repository. alt_tts_backends.py, tts_cache.py, and
KModelForONNX are fully written, well-organized, and never imported. The
fallback chain that the architecture diagram refers to has never been executed.
The child who chooses Swahili today is met with silence or broken audio not
because the answer doesn't exist in the code, but because it was never wired in.
The problem of implementing new G2P rule systems from scratch for a language
a contributor cannot natively evaluate is a research problem, not a 90-hour
engineering solution. Implementing phoneme rules incorrectly for Arabic or
Kinyarwanda without access to a native speaker for verification is not an option
because the output will be worse than the existing espeak fallback it is intended
to replace. The approach of this proposal is different. Integrate what exists,
measure what works, validate by listening to it, and document what doesn't so
that others can improve it. This is a promise of a solution that will work. The
quality of the pronunciation is data-driven, not data-based.

Technical Approach:

The Core Problem
Kokoro is a language-blind neural model at the acoustic level.It takes IPA
phonemes and a voice embedding as input and produces audio.
The voice embedding is the source of the sound of a language. This is the crucial
architectural point that informs the approach for all languages.
Languages with full Kokoro support (Spanish, French, Hindi, Portuguese,
Mandarin, Japanese, Italian, English)
The G2P routes and voice embeddings are already existing code. This is not
implementation; this is quality assurance.We want the pronunciation to be
phonetically correct and educationally sound, not just computationally correct.
Languages with G2P but no voice embedding (Arabic, Swahili,
Kinyarwanda)
For these languages, espeak-ng is able to output IPA phonemes.
However, Kokoro does not have a voice embedding model for any of them. The
process is cross-lingual transfer, where Arabic phonemes are passed through a
voice embedding model for a language such as Hindi.
The intelligibility of the output is up for empirical testing.
I will test all available voice embeddings against the scoring rubric and make the
backend decision based on the results in Week 4. If none of the embeddings have
a score above 3/5, Piper TTS is the backend for the language.
Languages with no coverage anywhere (Quechua, Guarani, Aymara)
These languages are not supported by Kokoro voice embeddings and, in general,
don't have good open-source TTS support.

How _resolve_requested_lang() works:
This function maps whatever voice is currently selected to a BCP-47 language
code for backend routing. It checks three things in order:
First, it looks for an explicit lang field in personas.json. Tier 3 personas like
Quechua have no Kokoro voice at all, so they carry their language directly:
{"name": "Inti", "lang": "qu", "backend": "mms"}. This bypasses all prefix logic.
Second, it extracts the Kokoro voice prefix and looks it up in the existing
VOICE_PREFIX_TO_LANG dict already in speech.py — af_ maps to en, hf_ to
hi, ef_ to es, and so on.
Third, if neither of those works, it returns "en" as a safe default. It never raises an
exception and never returns None.
The return value goes directly to get_tts_backend(lang), which checks the
preference array in alt_tts_backends.py and returns the highest-priority
available backend for that language.
This wiring also requires a new _stream_alt_audio() method in the speech.py
file, which will deal with the differing sample rates of the backends.
The sample rates are 24kHz for Kokoro, 22050Hz for Piper, and 16kHz for
MMS-TTS. The existing stream_kokoro_audio() function hardcodes the
GStreamer appsrc caps string as
'audio/x-raw,format=F32LE,rate=24000,channels=1'.
When 16kHz audio is provided, it will not only be distorted but can also trigger a
GStreamer negotiation error.
The new _stream_alt_audio() function will need to dynamically generate this
caps string from the reported sample rate of the backend, and I will check in
Week 5 whether autoaudiosink can deal with the differing sample rates internally
or whether an audioresample filter is required before the autoaudiosink.
Thread safety is managed through draining the current thread’s pipeline with an
EOS signal prior to switching, using a threading.Lock(). We’re switching between
utterances, never mid-utterance, so there’s no risk of tearing a buffer.
Before considering this integration finished, there’s a further set of corrections to
be made to the Piper voice IDs in alt_tts_backends.py. A few of those IDs are
outdated placeholders for voices that don’t actually exist in the
currentrhasspy/piper-voices registry.

6. Unicode and Script Specific Handling
To accommodate diverse scripts, specific handling strategies need to be
developed for text preprocessing, G2P translation, and UI display.
The strategy for handling different language groups is explained in detail in
the following points:

23

SugarLabs | Speak-AI Multilingual Support | GSoC’26

Text Normalization — applies to all languages before G2P
Children don't type clean prose. A normalize_text(text, lang_code)
function in speech_utils/normalizer.py will preprocess all input before it
reaches any G2P engine. Without this, input like "मेरा roll no. है 42" or "٣
‫ "طالب في الفصل‬reaches the G2P raw and produces wrong phonemes or silent
failures.
Hindi:
Arabic numerals expand to Hindi word form using num2words(lang='hi'),
NFC normalization applied, zero-width joiners stripped.
Arabic:
Arabic Indic numerals like ٣ expand to word form, convert Arabic Indic
numerals to Western digits, apply NFC normalization.
Quechua and Guarani:
NFC normalization is important because ejective apostrophes (qʼ, kʼ), and
tilde vowels can be represented as decomposed characters. Without NFC
normalization, the tokenizer will tokenize them incorrectly, and the G2P
output is garbage.
Fallback:
If num2words fails on unknown numerals, the raw digits pass through
unchanged. espeak-ng handles isolated digits acceptably in all languages
we support, so this will never cause silence.

Code-Switching Detection:
The "Hindi-English mixed" text "मेरा name है Rahul" is a very common
phenomenon in an Indian classroom, and this will be one of the most
frequent input types for a Hindi user. Processing mixed-script text through
a single G2P system will result in incorrect phonemes for the unsupported
script.
Detection approach: Unicode block majority-vote per word.

24

SugarLabs | Speak-AI Multilingual Support | GSoC’26

Words will be detected based on a majority vote of the dominant script.
"DEVANAGARI" will be mapped to "hi," "LATIN" to "en," and "ARABIC"
to "ar." Code segments with different scripts will be synthesized separately
using respective backends. The resulting audio arrays will be concatenated
with a 50ms silence inserted at each language boundary before being sent
to GStreamer.
Known limitation: English words written in Devanagari, such as "टे बल,"
will be sent to the Hindi backend.
This limitation is acceptable because espeak-ng works reasonably well for
common English loan words written in Devanagari.
Arabic (Right to Left):
Processing & Display: G2P translation in Espeak NG takes care of
right-to-left text handling at the phoneme level. GTK/Pango also supports
right-to-left text display.

Action Item: Confirm that the Gtk.Entry widget in activity.py changes
text direction correctly in response to Arabic text entry because this is not
necessarily automatic in all GTK configurations.
Hindi (Devanagari)
Known G2P Issue: The biggest issue with G2P translation in Espeak NG
for Hindi is incorrect Schwa deletion – i.e., deletion of inherent 'a' vowel in
specific phonological environments.
The fix is a word-final schwa deletion pre-processing step, implemented
in normalizer.py. It inserts a halant (U+094D, the explicit vowel
suppressor) before espeak-ng processes the text, based on a lookup against
a manually validated exceptions list. Luckily, as a native Hindi speaker, I
can construct and validate such a list myself, without a dataset to inform
me when "राम" should be /raːm/ and not /raːmə/.
Conservative approach: if we're unsure, we're not inserting a
halant. A wrong schwa will sound a bit odd, but will still be

25

SugarLabs | Speak-AI Multilingual Support | GSoC’26

understood. A wrong consonant will sound "broken." So, we
default to "do nothing."
This will be tested in Week 3's native speaker checkpoint with
our 18 sentence Hindi test corpus.

Mandarin (CJK Characters)
Processing: G2P translation in Mandarin is handled by misaki[zh].
Action Item: Testing will be restricted to tone sandhi – i.e., ensuring two
consecutive third tones correctly resolve to second-third tone – and
polyphonic character readings.
Quechua and Guarani (Extended Latin)
Processing: These scripts make heavy use of apostrophes for ejectives
and tildes for altered vowels.
Preprocessing Requirement: This script requires strict preprocessing
to normalize to NFC prior to tokenization. This is to prevent the tokenizer
from incorrectly splitting the text.

7.Offline First Model Management:
A ModelManager will be created based on this decision tree: check cache
for a model, try to download if a model is not cached, check SHA-256 hash
of what has beendownloaded, and use espeak with a notification to the user
if anything fails.
This ensures that speech will always be produced. There is never a
situation where a child will click on Speak and nothing will be produced.

The models will be downloaded lazily: only after a user selects a
language persona for the first time that requires a backend not cached
locally.
Downloads write to a .tmp path. After successful download, SHA-256 of
.tmp is verified against MANIFEST.json.

26

SugarLabs | Speak-AI Multilingual Support | GSoC’26

On successful download: atomic rename .tmp to .onnx. This is atomic
under Linux/POSIX: either the full verified model exists at the target path,
or it does not exist at all. There is no half-written model that the activity
could mistakenly read.
On failure or interruption: .tmp is immediately deleted, failure logged
to ~/.local/share/speak-ai/download_errors.log, and the activity reverts to
espeak-ng but does not crash. A single retry is scheduled for the next time
the activity is launched via a pending_downloads.json flag. The user sees a
GTK notification: "Voice download incomplete. Will retry when
connected."
Model versioning: MANIFEST.json is in the repository, and every model
is pinned to SHA-256 hash plus upstream version. Changing a model
requires a reviewed Pull Request. No auto-updating: a surprise 300MB
download during a lesson on a school machine is not acceptable.
Model versioning strategy: When a new voice pack is released by Piper
or Kokoro updates are released, ModelManager needs to be aware of this.
MANIFEST.json is checked into source control and specifies all models by
SHA-256 hash and upstream version. When a user wants to update a
model, a PR is created updating MANIFEST.json with the new hash and
URL. ModelManager will notice this on next launch and prompt the user to
update. This is a deliberate process and not automatic, which is good for
schools where bandwidth is limited and updating in the middle of a class is
a problem.

Technical Justifications
The section numbers below correspond to the content in "Proposed
Implementation" above.
2.1 Why the Alt Backend Wiring Matters
The first thing that struck me when reading the codebase is that
alt_tts_backends.py is fully implemented but never imported.
It includes MMSTTSBackend, PiperBackend, and get_tts_backend().

27

SugarLabs | Speak-AI Multilingual Support | GSoC’26

The fallback chain described in the proposal’s architecture never actually
runs. This isn’t a design gap; the modules are properly built. It’s a wiring
gap, and this is where the value of this project really lies.
The reason this matters so much is that Kokoro's approach is
language-agnostic at this acoustic level. It's taking in IPA phonemes and a
voice embedding and producing audio.
The voice embedding is what provides the acoustic characteristics of the
final output, not the language.
So, if I provide it with Arabic phonemes generated by espeak-ng and a
Spanish voice embedding, it will have no idea what Arabic is supposed to
sound like.
The result will not be Arabic spoken with a Spanish accent. The result will
be audio that doesn't even resemble a language. This is why the fallback
chain is not optional for Tier 2 and Tier 3 languages—it's the only option.

For the Quechua, Guarani, and Aymara languages, no Kokoro voice
embeddings are available at all.
Meta's MMS-TTS has VITS models available for these languages,
pre-trained on Bible recordings and similar data.
The naturalness is lower because the data is limited, but speech that is
intelligible at 2.5/5 is much more useful to a child in Cusco than speech
that is completely unintelligible.
For the Arabic and Swahili languages, the preferred Piper TTS is the
non-Kokoro fallback because it is ONNX-based, runs completely on CPU,
and has a maintained pack in the voice registry with file sizes listed.
The sample rate engineering problem is a real and non-trivial one.
Kokoro outputs at 24kHz. Piper outputs at 22050Hz. MMS-TTS outputs at
16kHz.
The current GStreamer pipeline in the stream_kokoro_audio() function
has the caps string hardcoded to 'rate=24000'.

28

SugarLabs | Speak-AI Multilingual Support | GSoC’26

If the function is given 16kHz input, it could either output pitch-shifted and
time-stretched audio or fail to negotiate the caps, depending upon the
behavior of the GStreamer library.
The _stream_alt_audio() function must construct the caps string from the
actual sample rate of the backend.
Whether or not autoaudiosink performs resampling internally will be
verified in Week 5, as it depends upon the version of the plugin present in
the target system.
Thread safety is achieved by draining the active pipeline using EOS before
switching and protecting the pipeline reference using a threading.Lock().
Since language switching only occurs between utterances, there is never an
active audio buffer that is split mid-stream.

2.2 Why the Cache Is Not Optional
The tts_cache.py module is fully implemented with correct LRU logic, SHA-256
keys, and disk eviction, but is currently not integrated with the rest of the system.
The price of this disconnection is paid in full by every child using this activity on
classroom machines.
A 10-year-old child learning to read does not type a sentence, hear it, and then go
on to another activity.
No, she types a word, hears it, changes a letter, and then hears it again. This
small phrase is repeated five or six times in two minutes.
The current system forces a full Kokoro inference on every single repetition:
phonemes through the neural network, audio tensors, and GStreamer streaming.
The wiring of this cache completely eliminates the repetition cost. The key
generation using SHA-256 hash on the (text, voice, lang_code, speed) tuple
incurs a completely negligible 0.003 ms cost. The cost of resolving a cache hit,
which loads a numpy array from local storage, is effectively zero.
Across different audio clip lengths, we measured:

29

SugarLabs | Speak-AI Multilingual Support | GSoC’26

0.5-second audio clip (47KB): p50 latency 0.21 ms
3.0-second audio clip (281KB): p50 latency 0.23 ms
8.0-second audio clip (750KB): p50 latency 0.33 ms
The critical piece is that a warm kokoro synthesis on x86_64 takes 1729 ms. By
serving this audio from the cache, which takes on average 0.25 ms, we get a
6915x speedup on repeated phrases. On ARM-based classroom machines, where
kokoro synthesis is significantly slower, this will be a higher number.

Additionally, with this LRU cache, we are assured that disk usage will always be
strictly bounded (capped at 500 entries or 200 MB), which is a safe choice given
many older classroom machines will have little storage capacity.
This takes several seconds per repetition, which makes this interaction look like
a failure, not a learning tool.

2.3 Testing Cross-Lingual Transfer Before Giving Up on
Kokoro
For Arabic, Swahili, and Kinyarwanda, there is no trained Kokoro voice
embedding.
Fortunately, Kokoro's acoustic model is language-agnostic: it takes any
string of IPA phonemes and maps it to audio via a voice embedding. The
input to the model and the embedding are independent. This is an
empirical question worth answering before we move on to using Piper:
do we get recognizable Arabic audio if we give it a string of correct Arabic
IPA and a Hindi voice embedding, since we know that espeak-ng will
produce that string?
This has not been tested for Arabic or Swahili, though. This will be tested
in Week 4 by generating test audio for each of the three languages using
every Kokoro voice family, scoring each combination using the
pronunciation rubric, and making backend decisions based on actual score.
If at least one voice family has a score over 3/5, then Kokoro is the backend
for that language, along with the best-performing embedding family.

30

SugarLabs | Speak-AI Multilingual Support | GSoC’26

If none do, then the preference array in `alt_tts_backends.py` is adjusted
from `['kokoro', 'piper', 'mms']` to `['piper', 'kokoro', 'mms']` for that
language.
This is a change to the configuration, not the logic.
The motivation for testing this is efficiency. If cross-lingual Kokoro transfer
is good enough for Swahili,
then we don't need to add a dependency on Piper or download a 60MB
voice pack for that language.
If it is not, then we have hard data on why, not just an assumption.

2.4 The ONNX Export Path
The target is to achieve a CPU speedup of 2 to 3 times on the hardware in
the classroom – ARM devices without a GPU.
However, there is a blocker that has so far not allowed this to be done:
torch.istft in the iSTFTNet decoder in Kokoro.
ONNX tracing cannot trace complex numbers, but torch.istft depends on
complex numbers. However, this issue is already half-solved in the code:
KModel takes an argument disable_complex=True, in which case it
replaces TorchSTFT with CustomSTFT.
This is because CustomSTFT computes the exact same thing using
F.conv1d and F.conv_transpose1d, which ONNX tracing supports.
There is already a model in model.py: KModelForONNX.
But there is no code to write an ONNX export script – this is the 8 hours to
complete in Week 8.
The go/no-go with the mentors is at the end of Week 8.
However, in case tracing fails because CustomSTFT is not enough – in case
other operations in the model are not traceable – then those 8 hours will be
used on pronunciation quality, since this is more important per project
description.

2.6 Why the Mouth Animation Does Not Break
The mouth animation subsystem reads S16LE 16-bit PCM audio from the
fakesink branch of the GStreamer pipeline, which is derived from the peak
amplitude.
Kokoro produces F32LE 32-bit audio. If 32-bit audio were to be used by
the animation code, the byte offsets would be incorrect, resulting in
random mouth animation with no correlation to the speech.
The GStreamer pipeline already addresses this issue in a correct manner.
The pipeline contains a tee element, which splits the signal into two
branches.
One branch is sent directly to autoaudiosink at 24kHz F32LE for playing,
and the other branch is sent through an audioconvert element followed by
an audioresample element before going to the fakesink at 16kHz S16LE for
animation. The animation code only receives well-formatted 16-bit data,
which does not require any changes.
For the alternative backends, which produce 16kHz or 22050Hz audio, the
same branch of the pipeline addresses the format difference. The
audioresample element resamples to 16kHz regardless of what frequency
the input audio was sampled at.

Addressing Hardware Constraints: RAM and Cold Starts:
The profiling data from the measurements above indicates two key constraints:
1.33 GB RSS Memory Footprint, as well as a Cold Start of approximately 44
seconds, including disk I/O operations and model loading.
1)​ For the 1 GB RAM Device Mitigation, ModelManager will use the
psutil library to detect available system memory on initialization. For
devices with 1 GB of RAM (e.g., older XO laptops), loading Kokoro or
MMS will cause an Out of Memory (OOM) crash. For such devices, heavy
AI backends are disabled, and the activity defaults to the lightweight
espeak-ng engine only.

SugarLabs | Speak-AI Multilingual Support | GSoC’26

2)​ UX During Cold Starts: For devices with sufficient RAM (e.g., 2 GB
RAM + RPi4) but slow CPU/SD cards, waiting 44 seconds for the disk I/O
operations to complete is unpalatable for a child. For this, ModelManager
loads the heavy TTS engines asynchronously in the background. When a
user wants to hear an audio prompt, ModelManager immediately starts
playing synchronous espeak-ng audio if Kokoro is still loading in the
background. When the asynchronous loading of Kokoro in the background
is complete, the user can seamlessly transition to the high-quality
Kokoro/Piper voice for subsequent audio prompts.

Objective Benchmarking: Phoneme Error Rate (PER)
Analysis:
Although human evaluation rubrics are essential for naturalness evaluation,
Speak-AI also demands a deterministic, mathematically verifiable metric to
ensure that there are no regressions in phonetics. For this purpose, I will add an
automated Phoneme Error Rate (PER) calculation pipeline for each of the
languages.
The Validation Harness: I will create a specific script for validation purposes
(tests/eval_{lang}.py).
The Calculation: The script will use the newly wired G2P modules on a curated
100-word validation corpus for each target language. It will then use the
calculated Levenshtein Distance on the generated IPA transcripts against a
reference dictionary to calculate the PER.
The Success Threshold: The specific engineering goal is to achieve a PER of
<15% against the eSpeak-ng baseline for all Tier 1 languages. By hardcoding this
into the test suite, we can ensure that each of the newly merged language
modules brings a quantifiable improvement in pronunciation accuracy before it
ever sees a student's XO laptop.
